# 开发手册

## 1. 项目概述

**Token Cost App - OC Codex** 是一个 macOS 原生桌面应用，用于可视化 AI 编程工具（OpenCode / Codex）产生的 token 用量和费用数据。

- **语言**: Swift 6.0+
- **UI 框架**: SwiftUI + AppKit (部分)
- **构建系统**: Swift Package Manager (SPM)
- **最低系统**: macOS 14.0 (Sonoma)
- **仓库**: [GitHub](#) · MIT License

### 数据源

| 数据源 | 格式 | 默认路径 |
|--------|------|---------|
| **OpenCode** | SQLite 数据库 | `~/.local/share/opencode/opencode.db` |
| **Codex** | JSONL 文本文件 | `~/.codex/sessions/` |

## 3. 模块架构

### 3.1 CodexTokenCostCore（核心库）

最底层模块，不依赖 AppKit / SwiftUI。被 App 和 Helper 共享引用。

| 文件 | 职责 |
|------|------|
| `Models.swift` | 数据模型：`TokenCostSource`, `TokenCostSettings`, `DashboardPayload`, `UsageAggregateRow` |
| `AppPreferences.swift` | 全局偏好：语言、总览计价模式、Provider 计费方案选择、本地持久化 |
| `BillingPlanCatalog.swift` | Provider 计费计划目录：OpenCode、ChatGPT/Codex、MiniMax、Xiaomi MiMo 官方预设与自定义 USD 月费解析 |
| `Localization.swift` | 语言切换与本地化资源加载 |
| `CodexModels.swift` | Codex 专用模型：`CodexTokenUsage`, `CodexSessionSummary`, `CodexDashboardPayload` |
| `DashboardAnalytics.swift` | 分析引擎：将原始行转为 Overview / Cache / Provider Rank / Model Comparison / Distribution / Trend / Stacked。**含模型定价目录**（`TokenCostPricingCatalog`） |
| `CodexAnalytics.swift` | Codex 分析：session 排序、日期桶聚合、趋势点生成 |
| `CodexBilling.swift` | Codex 默认计费桥接（从 `BillingPlanCatalog` 读取 ChatGPT Plus 口径） |
| `TokenDatabaseClient.swift` | SQLite 客户端：探针 + `json_extract` 查询 + 构建 `DashboardPayload` |
| `SourceDiscoveryService.swift` | 文件系统扫描：遍历目录发现数据库文件。`lastModified` 字段以 ISO 8601 UTC 格式存储，UI 层通过 `TokenCostFormatters.localDateTime(_:)` 转换为用户系统时区显示 |
| `CodexSessionCollector.swift` | JSONL session 解析器：读取每行 JSON、提取 `session_meta` 和 `event_msg(token_count)` |
| `SourceProfiles.swift` | OpenCode / Codex 的默认路径、文件模式、扫描深度等配置 |
| `SettingsStore.swift` | 设置 + 快照文件的读写和轮转 |
| `SafeFileStore.swift` | 沙箱内安全文件操作（防止路径穿越） |
| `UpdateChecker.swift` | GitHub Release 检测、.zip 下载（异步进度回调）、完整性校验、ditto 解压 |
| `PathUtilities.swift` | 路径标准化、descent 检查 |
| `AppPaths.swift` / `CodexSessionPaths.swift` | 路径常量 |
| `Theme.swift` | 主题枚举 |

### 3.2 CodexTokenCostApp（主应用）

依赖 Core 模块。`@main` 入口。

| 目录 | 职责 |
|------|------|
| `App/` | `@main` 入口、AppDelegate、路径常量、菜单命令 |
| `Stores/` | `TokenCostModel`（OpenCode 数据层）、`CodexSessionModel`（Codex 数据层）、`AppPreferencesModel`（全局偏好桥接）、`UpdateCheckerModel`（版本更新状态机）、`CodexSnapshotStore` |
| `Views/` | 所有 SwiftUI 视图（6 个主要页面 + 通用组件）。`Components.swift` 含 `TokenCostFormatters` 枚举（token/currency/percent/localDateTime 格式化）和 `TokenMetricCard`（指标卡片组件，支持 compact 模式和统一高度约束） |
| `Services/` | `CodexHelperRunner`（通过 `Process` 调起 Helper 子进程） |
| `Support/` | `ThemePalette`（主题调色板）、`SeriesPalette`（图表配色） |
| `Resources/` | App target 内置资源，例如 `Pricing.md` 只读计费参考文档 |

### 3.3 CodexTokenCostHelper（辅助进程）

独立可执行文件，由主 App 通过子进程方式调用。

| 文件 | 职责 |
|------|------|
| `main.swift` | CLI 入口：解析 `--source-root` / `--manual-source-path` 参数 |
| `CodexSessionCollector.swift` | 采集 Codex session 文件，解析为 `CodexDashboardPayload`，输出 JSON 到 stdout |

### 3.4 应用级偏好与本地化

应用级偏好不和 OpenCode / Codex 各自的数据源设置混用，而是单独保存：

- `AppPreferencesStore` 负责把 `language`、`openCodePricingMode` 与 `billingSelectionsByProvider` 写入本地 `config/app-preferences.json`
- `AppPreferencesModel` 是 SwiftUI 侧的单一状态入口，负责保存和广播语言切换
- `AppLocalization` 从 `Resources/<lang>.lproj/Localizable.strings` 读取文案，并在 App 运行时根据语言切换刷新 UI
- 总览页只读取 `openCodePricingMode`；其中 OpenCode 的“实际 token”指标按 `input + output + reasoning` 计算（各字段含义与 OpenCode 源码 `getUsage()` 存储逻辑一致：`input` 在 SQLite 中已是非缓存输入，`output` 为非推理输出，`reasoning` 为推理 token，`cacheRead`/`cacheWrite` 为独立缓存字段），OpenCode / Codex 独立页面仍保留各自原有 token 统计与展示口径
- Provider 订阅成本通过 `BillingPlanCatalog` 解析：未保存自定义选择时回退到现有默认费用（OpenCode Go $10、ChatGPT Plus $20、MiniMax Plus 极速版 ¥98、Xiaomi MiMo 当前默认 ¥34.9）；选择 DIY 时统一按 USD/月保存并注入 `TokenCostDashboardAnalytics` 的 provider effective cost
- `PricingDocView` 从 `Bundle.module` 读取 app target 内置的 `Pricing.md`，手动分段解析标题、引用块和表格，过滤 markdown 分隔符行（`|---|---|`），使用原生 `HStack` + `Divider` 渲染表格；不打开任意本地路径、不访问网络；`Package.swift` 通过 `.process("Resources")` 打包该资源
- 技术词统一保留英文写法，例如 `token`、`API`、`OpenCode`、`Codex`、`ChatGPT Plus`

## 4. 数据流

### 4.1 OpenCode 数据管线

```
[文件系统扫描]
SourceDiscoveryService
    ↓ discover()
[SQLite 数据库列表] → TokenCostSource[]
    ↓ 用户选择来源
[TokenDatabaseClient.loadPayload()]
    ↓ SELECT json_extract(...) FROM message
[UsageAggregateRow[]]
    ↓ buildPayload()
[DashboardPayload]
    ↓ TokenCostDashboardAnalytics 初始化（可接收 billingOverridesByProviderKey）
[Overview / Cache / Provider / Model / Distribution / Trend]
    ↓ 各 View 读取
[UI 渲染]
```

**关键 SQL**: 使用 `json_extract(data, '$.tokens.input')` 等从 JSON 列提取字段，按 `date, model_id, provider_id` 分组聚合。

### 4.2 Codex 数据管线

```
[用户在设置中配置 session 目录/文件]
    ↓
[CodexHelperRunner.loadPayload()]
    ↓ 启动子进程
[CodexTokenCostHelper (外部进程)]
    ↓ 遍历 JSONL 文件
[CodexSessionCollector.loadPayload()]
    ↓ 逐行解析 session_meta / event_msg
[CodexDashboardPayload]
    ↓ stdout JSON → 主进程解码
[CodexSessionModel.payload]
    ↓ CodexDashboardAnalytics 分析
[UI 渲染: 趋势图 + Session 列表]
```

**Error 处理**: Helper 进程超时 15s，失败自动回退到本地快照 `~/Library/Application Support/com.yanghaoran.CodexTokenCost/snapshots/codex/`。

## 5. 编译、构建与打包

### 5.1 编译验证

| 命令 | 作用 | 产物 |
|------|------|------|
| `swift build` | Debug 编译验证 | `.build/` |
| `swift build -c release` | Release 编译验证 | `.build/` |

这两条命令只负责编译，不承诺 `release/` 下的发布目录。

> 备注：如果当前 shell 访问 `~/.cache/clang/ModuleCache` 受限，可以改用 `HOME=/private/tmp swift build --disable-sandbox`；`script/build_and_run_codex.sh` 已内置这条兜底路径。

### 5.2 本地构建 app

```bash
bash script/build_and_run_codex.sh build
```

- 版本号默认读取最近的 git tag，格式为 `vX.Y.Z`
- 本地构建必须落在 `release/` 下的独立子目录中，目录名固定为 `release/vX.Y.Z-YYYYMMDD-HHMMSS-<pid>/`
- 这个本地子目录只保存当前开发状态的 app bundle，不覆盖正式 release 目录
- 生成的 app bundle 名称固定为 `Token Cost App - OC Codex.app`
- 构建脚本会把 `Resources/` 整体复制进 `Contents/Resources/`，所以 `zh-Hans.lproj` 和 `en.lproj` 会随 app 一起打包
- SPM app target 也会通过 `.process("Resources")` 打包 `Sources/CodexTokenCostApp/Resources/Pricing.md`，供 `PricingDocView` 在 app 内只读展示
- `CFBundleDevelopmentRegion` 设为 `zh-Hans`，默认文案基准是中文
- `run`、`debug`、`logs`、`telemetry`、`verify` 也使用同一套本地 staging 命名

### 5.3 正式 Release 打包

```bash
RELEASE_VERSION=v0.1.1 bash script/build_and_run_codex.sh release
```

- 当前已发布版本是 `v0.1.0`；如果还没有新的 tag，下一版继续按 `v0.1.1` 维护在 `Unreleased`，并保留相对 `v0.1.0` 的累计变更。
- 正式 release 目录固定为 `release/vX.Y.Z/`
- 同目录下生成 `Token Cost App - OC Codex.zip`
- GitHub Actions 的 `release.yml` 会注入当前 tag 作为 `RELEASE_VERSION`

### 5.4 依赖

- macOS 14.0+
- Xcode / Command Line Tools (提供 `xcrun`, `swift`, `lldb`)
- SQLite3 (系统自带)
- 无其他外部依赖

## 6. 版本与命名规则

### 语义化版本 (SemVer)

```
MAJOR.MINOR.PATCH
0.1.1
```

| 变更类型 | 版本 | Git 操作 |
|---------|------|---------|
| 大重构 / 不兼容变更 | MAJOR | `git tag -a v1.0.0 -m "..."` |
| 新功能 | MINOR | `git tag -a v0.3.0 -m "..."` |
| Bug 修复 | PATCH | `git tag -a v0.2.1 -m "..."` |

### 6.1 命名矩阵

| 场景 | 命名规则 | 说明 |
|------|----------|------|
| 编译验证 | `.build/` | 只做编译校验，不生成发布目录 |
| 本地构建 app | `release/vX.Y.Z-YYYYMMDD-HHMMSS-<pid>/` | `vX.Y.Z` 取最近 git tag，时间戳和 PID 只用于避免覆盖 |
| 正式 release | `release/vX.Y.Z/` | 只保留版本号，不混入时间戳 |

### 6.2 发布产物

- app bundle 名称固定为 `Token Cost App - OC Codex.app`
- 正式 release 的 zip 与 app bundle 同目录保存
- `release/latest/` 始终指向最新构建产物（.app 和 .zip），所有构建模式（run/build/release）均会更新
- `release/versions.json` 记录正式 release 的结构化元数据（版本号、日期、文件名），仅 release 模式写入
- release workflow 只上传版本化 zip

### 发布流程

```
1. 更新 CHANGELOG.md 和 SECURITY.md（写入本次变更并同步支持版本）
2. 打 tag：`git tag -a v0.1.1 -m "Release notes"`
3. 推送 tag：`git push origin v0.1.1`
4. GitHub Actions (release.yml) 自动触发：
   ├── `RELEASE_VERSION=$GITHUB_REF_NAME bash script/build_and_run_codex.sh release`
    └── 创建 GitHub Release + 上传 `release/vX.Y.Z/Token Cost App - OC Codex.zip`
5. 在 GitHub Releases 页面补充 Release Notes
```

### 6.3 发版同步要求

- 每次准备发布前，必须同步检查并更新 `CHANGELOG.md` 与 `SECURITY.md`
- `CHANGELOG.md` 负责记录本次变更和对应版本号，`SECURITY.md` 负责记录当前支持的安全版本范围
- 在未打正式 tag 之前，不要把 `Unreleased` 写成已发布版本；`Unreleased` 里应保留相对当前稳定版的累计变更，等 `vX.Y.Z` tag 推送后，再把对应条目定稿为正式 release
- 若发布版本号发生变化，`script/build_and_run_codex.sh`、GitHub Release 目录和 zip 命名必须保持一致
- 清理 `release/` 目录时，至少保留 1 份正式 release 和 1 份独立的最新带时间戳本地构建快照；旧快照再移动到垃圾桶。`release/latest/`、`release/versions.json` 和 `release/release-notes/` 不可删除

遵循 [Keep a Changelog](https://keepachangelog.com/)：

## 7. 扩展指南

### 7.1 如何添加新数据源

1. 在 `Models.swift` 的 `TokenCostSourceFamily` 枚举中添加新 case
2. 在 `SourceProfiles.swift` 添加新 profile（默认路径、文件模式等）
3. 实现新的 Client（类似 `TokenDatabaseClient`）
4. 在 `TokenCostModel` 或新建 Store 中集成
5. 添加新的 View Page

### 7.2 如何添加新的分析卡片

1. 在 `TokenCostDashboardAnalytics` 或 `CodexDashboardAnalytics` 中添加新的结构体和计算属性
2. 在对应 PageView 中添加 `TokenSectionCard` + `TokenMetricCard`

### 7.3 如何添加新主题

1. 在 `Theme.swift` 的 `TokenCostThemeChoice` 枚举中添加 case
2. 在 `ThemePalette.swift` 的 `init(theme:)` 中添加颜色定义

### 7.4 版本更新检查机制

更新检查由 Service + Store 两层协作完成：

| 层 | 文件 | 职责 |
|----|------|------|
| Service | `UpdateChecker.swift` | GitHub API 调用、版本比较、下载（异步 `URLSession.bytes` 带回调进度）、ditto 解压、文件完整性校验 |
| Store | `UpdateCheckerModel.swift` | `@MainActor ObservableObject`，管理状态机（idle → checking → updateAvailable → downloading → downloadComplete → error），协调检查/下载/安装流程 |

**数据流**：
```
[App 启动 .task]
    ↓ checkForUpdate()
[读取本地缓存 update-check.json]
    ↓ 24h 内？→ 用缓存结果
    ↓ 超时？→ GET api.github.com/repos/.../releases/latest
[返回 GitHubRelease JSON]
    ↓ semverCompare(latest, current)
[有更新 → state = .updateAvailable]
    ↓ 用户点击"更新"
[startDownload() → URLSession.bytes → 进度回调 → 进度 fill 动画]
    ↓ 下载完成
[校验文件大小 × Content-Length 交叉验证]
    ↓ ditto -x -k 解压
[state = .downloadComplete → 用户点击"安装" → NSWorkspace.open(.app)]
```

- 下载文件存放于 `Application Support/com.yanghaoran.CodexTokenCost/updates/`
- 检查时间戳缓存在 `config/update-check.json`，24 小时内不重复请求
- 所有网络请求使用 `URLSession(configuration: .ephemeral)`，无持久化 cookie/cache

## 8. 安全边界

- 本应用**只读**访问 OpenCode 数据库和 Codex session 文件
- 不修改任何源数据文件
- 语言设置和总览计价偏好仅写入本地 `Application Support` 下的 `config/app-preferences.json`
- 总览页的实际 token 仅由现有来源数据派生，不会回写到数据库、session 文件或网络
- 本地化资源只读加载，不会反向写回源数据或网络
- 配置和快照仅写入 `~/Library/Application Support/com.yanghaoran.CodexTokenCost/`
- Helper 子进程通过标准 OS 进程隔离运行
- 文件写入通过 `SafeFileStore` 防止路径穿越
- 无网络请求、无数据上传——除可选手动触发的 GitHub Release 版本更新检查（仅匿名 GET `api.github.com`，不传数据）和可选手动开启的 Provider 余额查询（HTTPS 直连各 Provider 官方 API）外
